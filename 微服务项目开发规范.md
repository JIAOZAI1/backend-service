# 微服务项目开发规范

> 适用范围：本规范适用于公司内部所有微服务项目、公共组件库、运维脚本、CI/CD 配置及相关文档。
>
> 目标：统一项目结构、命名方式、交付标准和协作流程，降低新人接入成本，提升多团队协作效率。

---

# 1. 规范目标

本规范用于统一以下内容：

* 项目根目录结构
* 微服务目录结构
* 服务命名规范
* 文件与目录命名规范
* 多语言服务组织方式
* 运维脚本规范
* 部署目录规范
* 配置文件规范
* API、数据库、分支、提交规范
* 新服务接入标准

# 2. 总体原则

## 2.1 统一仓库模式

默认采用 **Monorepo** 模式管理整个微服务解决方案，所有服务、公共包、部署配置、脚本、文档均放在同一个仓库中。

## 2.2 分层清晰

项目按职责分层：

* `services/`：业务微服务
* `packages/`：公共库、SDK、基础组件
* `deploy/`：部署相关配置
* `scripts/`：运维/构建/开发脚本
* `configs/`：全局配置模板
* `docs/`：规范文档、架构文档
* `tests/`：跨服务集成测试 / E2E 测试

## 2.3 命名统一

统一采用**小写短横线命名（kebab-case）**，避免大小写、下划线、拼音缩写混用。

## 2.4 服务自治

每个微服务应具备以下能力：

* 独立启动
* 独立测试
* 独立构建
* 独立部署
* 独立文档
* 独立配置

# 4. 目录命名规范

## 4.1 命名格式

所有目录统一使用：

```text
kebab-case
```

示例：

```bash
user-service
order-service
payment-service
build-image
deploy-k8s
database-backup
```

禁止使用：

```bash
UserService
user_service
userService
userservice
```

## 4.2 命名原则

* 名称应表达**业务含义**或**职责**
* 避免无意义缩写
* 避免技术细节侵入业务目录名
* 避免中文目录名

# 6. 服务命名规范

## 6.1 服务命名格式

统一格式：

```text
<domain>-service
```

示例：

* `user-service`
* `order-service`
* `payment-service`
* `product-service`
* `inventory-service`
* `notification-service`
* `auth-service`
* `file-service`

## 6.2 服务命名原则

服务名称应表达**业务边界**，而不是技术实现。

正确示例：

* `order-service`
* `payment-service`
* `inventory-service`

错误示例：

* `spring-order-service`
* `mysql-user-service`
* `user-api-service`
* `common-service`

## 6.3 公共能力服务命名建议


| 领域 | 服务名                 |
| ---- | ---------------------- |
| 用户 | `user-service`         |
| 认证 | `auth-service`         |
| 订单 | `order-service`        |
| 支付 | `payment-service`      |
| 商品 | `product-service`      |
| 库存 | `inventory-service`    |
| 通知 | `notification-service` |
| 文件 | `file-service`         |
| 搜索 | `search-service`       |
| 报表 | `report-service`       |

## 6.4 租户服务命名规范

租户维度的业务服务（即需要直连某个租户独立数据库、按 `tenantCode` 区分数据源的服务）统一命名为：

```text
tenant-<业务名>-service
```

示例：

* `tenant-order-service`
* `tenant-report-service`
* `tenant-billing-service`

与普通 `<domain>-service` 的区别：`tenant-` 前缀表明该服务的数据访问层需要通过 [`packages/tenant-db-connector`](#102-已有公共包) 按 `tenantCode` 动态获取数据库连接（而不是像 `sso-service`/`admin-service` 那样连接固定的单一数据库），便于从服务名直接识别其数据访问模式。不满足这一特征的服务（即使业务上服务于多租户场景，但数据库连接是固定的）不使用此前缀，仍按 6.1 节 `<domain>-service` 命名。

# 7. 多语言服务规范

允许不同服务使用不同语言，但要求**外层目录结构一致**，服务内部遵循该语言生态的主流规范。

## 7.1 Go 服务结构

```bash
services/order-service/
├── cmd/
│   └── server/
│       └── main.go
├── internal/
│   ├── handler/          # HTTP/gRPC 入口，参数解析与响应封装
│   ├── service/           # 业务逻辑编排
│   ├── repository/        # 数据访问接口与实现
│   └── model/             # 领域模型 / DTO
├── pkg/                    # 可被外部安全引用的公共代码
├── tests/
├── configs/
├── migrations/
├── go.mod
├── go.sum
├── Dockerfile
├── Makefile
└── README.md
```

### 7.1.1 分层依赖方向

```text
handler → service → repository → model
```

* `handler` 不写业务逻辑，只做参数校验、调用 `service`、组装响应
* `service` 通过接口依赖 `repository`，不直接引用具体数据库驱动
* `internal/` 下的包只允许被本服务引用，禁止被其他服务 import；跨服务共享代码必须下沉到 `packages/`
* `pkg/` 仅存放明确设计为可复用、无内部业务耦合的代码

### 7.1.2 命名与代码风格

* 包名使用简短小写单词，不使用下划线或驼峰，如 `handler`、`repository`
* 导出类型、函数使用 **PascalCase**（如 `OrderService`），非导出使用 **camelCase**（如 `orderRepo`）
* 接口命名简洁，通常以行为或角色命名（如 `OrderRepository`），不强制 `I` 前缀
* 统一使用 `gofmt` / `goimports` 格式化，CI 中执行 `golangci-lint run` 作为强制检查
* 错误处理统一使用 `errors.Is` / `errors.As`，禁止吞掉错误（`_ = err`）

### 7.1.3 配置与依赖注入

* 配置统一通过 `configs/app.<env>.yaml` 加载，推荐使用 `viper` 或团队统一的配置库
* 依赖组装（DB、Redis、MQ 客户端等）统一在 `cmd/server/main.go` 或 `internal/wire` 中完成，推荐使用构造函数注入，避免全局变量
* 禁止在 `service` / `repository` 包中读取环境变量或配置文件，配置应通过参数或结构体传入

### 7.1.4 数据访问

* 数据库访问统一封装在 `internal/repository/`，对上层暴露接口而非具体实现（便于 mock 测试）
* 推荐使用 `sqlx` / `gorm` / `ent` 等团队统一选型，禁止在 `handler` 或 `service` 中直接拼接 SQL
* 数据库迁移统一使用 `golang-migrate` 或团队统一工具，迁移文件放在 `migrations/`，禁止手工改表

### 7.1.5 测试

* 单元测试文件与被测文件同目录，命名为 `xxx_test.go`
* 使用标准库 `testing`，断言库推荐 `testify/assert` 或 `testify/require`
* `repository` 层测试可使用接口 mock（如 `mockery` 生成），`service` 层测试禁止连接真实数据库
* 集成测试统一放在 `tests/`，通过 `docker-compose` 或 `testcontainers-go` 启动真实依赖

### 7.1.6 服务级 Makefile 建议目标

```makefile
run              # go run ./cmd/server
test             # go test ./... -race -cover
lint             # golangci-lint run
build            # go build -o bin/server ./cmd/server
docker-build     # docker build -t order-service .
migrate-up       # migrate -path migrations -database $(DB_URL) up
migrate-down     # migrate -path migrations -database $(DB_URL) down 1
```

## 7.2 Node.js / TypeScript 服务结构

```bash
services/user-service/
├── src/
│   ├── controllers/       # 路由入口，参数校验与响应封装
│   ├── services/           # 业务逻辑编排
│   ├── repositories/        # 数据访问层
│   ├── models/              # 实体 / DTO / Schema
│   ├── middlewares/         # 鉴权、日志、错误处理等中间件
│   ├── config/               # 配置加载
│   └── main.ts
├── tests/
│   ├── unit/
│   └── integration/
├── configs/
├── migrations/
├── package.json
├── tsconfig.json
├── .eslintrc.js
├── .prettierrc
├── Dockerfile
├── Makefile
└── README.md
```

### 7.2.1 分层依赖方向

```text
controllers → services → repositories → models
```

* `controllers` 不写业务逻辑，只做参数校验（推荐 `zod` / `class-validator`）、调用 `services`、返回响应
* `services` 通过接口/抽象类依赖 `repositories`，不直接引用 ORM 客户端实例
* 跨服务共享代码下沉到 `packages/`，禁止服务间直接相互 import

### 7.2.2 命名与代码风格

* 强制使用 **TypeScript**，禁止新服务使用纯 JavaScript
* 文件名统一使用 kebab-case（如 `order-service.ts`），类名使用 **PascalCase**，变量/函数使用 **camelCase**
* 接口以 `I` 前缀或语义化命名均可，团队内需统一，禁止混用
* 统一使用 ESLint + Prettier，CI 中执行 `npm run lint` 与 `npm run format:check` 作为强制检查
* 异步统一使用 `async/await`，禁止裸 `.then()` 链式调用超过两层

### 7.2.3 配置与依赖注入

* 配置通过 `dotenv` 加载 `.env.<env>` 文件，或使用 `@nestjs/config` 等团队统一方案
* 推荐使用依赖注入框架（如 NestJS 内置 DI）管理 `services` 与 `repositories` 的生命周期，避免手工 `new` 散落各处
* 禁止在 `controllers` 中直接读取 `process.env`

### 7.2.4 数据访问

* 统一使用团队选型的 ORM（如 `TypeORM` / `Prisma` / `Sequelize`），封装在 `repositories/` 中
* 数据库迁移统一使用 ORM 自带迁移工具生成，放在 `migrations/`，禁止直接改表
* `services` 层禁止拼接原生 SQL，确需原生 SQL 时必须通过 `repositories` 层的参数化查询

### 7.2.5 测试

* 单元测试使用 `Jest`（或团队统一框架），文件命名 `*.spec.ts`，与源文件同目录或放在 `tests/unit/`
* 集成测试放在 `tests/integration/`，通过 `supertest` 驱动 HTTP 层，数据库依赖通过 `testcontainers` 或 Docker Compose 启动
* Mock 第三方服务统一使用 `jest.mock` 或 `nock`，禁止单元测试中发起真实网络请求

### 7.2.6 服务级 Makefile 建议目标

```makefile
run              # npm run start:dev
test             # npm run test
lint             # npm run lint
build            # npm run build
docker-build     # docker build -t user-service .
migrate-up       # npm run migration:run
migrate-down     # npm run migration:revert
```

## 7.3 Java 服务结构

```bash
services/payment-service/
├── src/
│   ├── main/
│   │   ├── java/com/company/paymentservice/
│   │   │   ├── controller/       # REST 入口，参数校验与响应封装
│   │   │   ├── service/           # 业务逻辑接口与实现
│   │   │   ├── repository/        # 数据访问层（Spring Data / MyBatis）
│   │   │   ├── domain/             # 实体 / 值对象
│   │   │   ├── dto/                 # 请求 / 响应 DTO
│   │   │   ├── config/               # 配置类、Bean 定义
│   │   │   └── PaymentServiceApplication.java
│   │   └── resources/
│   │       ├── application.yaml
│   │       └── application-<env>.yaml
│   └── test/
│       └── java/com/company/paymentservice/
├── configs/
├── migrations/
├── pom.xml            # 或 build.gradle
├── Dockerfile
├── Makefile
└── README.md
```

### 7.3.1 分层依赖方向

```text
controller → service → repository → domain
```

* `controller` 只做参数校验（`@Valid` + Bean Validation）、调用 `service`、返回统一响应体
* `service` 面向接口编程，注入 `repository` 接口而非实现类，业务逻辑不感知底层持久化细节
* `domain` 实体不依赖 `controller` / `service`，禁止在实体类上编写业务逻辑之外的框架耦合代码

### 7.3.2 命名与代码风格

* 包名统一小写，格式为 `com.<company>.<service>`，如 `com.company.paymentservice`
* 类名使用 **PascalCase**（如 `PaymentService`），方法/变量使用 **camelCase**
* 接口与实现类命名：接口 `PaymentService`，实现类 `PaymentServiceImpl`（Spring 生态惯例）
* 统一使用 Checkstyle / Spotless 做代码风格检查，CI 中强制执行 `mvn spotless:check`（或 `./gradlew spotlessCheck`）
* 禁止使用通配符导入（`import xxx.*`）

### 7.3.3 配置与依赖注入

* 统一使用 Spring Boot 的 `application-<env>.yaml`，环境名遵循本规范第 14 章
* 依赖注入统一使用构造器注入（`@RequiredArgsConstructor` 或显式构造函数），禁止使用字段注入（`@Autowired` 标注字段）
* 敏感配置通过环境变量或 Spring Cloud Config / Vault 注入，禁止硬编码在 `application.yaml`

### 7.3.4 数据访问

* 统一使用 `Spring Data JPA` 或 `MyBatis`（团队内二选一，不混用），仓储接口放在 `repository/`
* 数据库迁移统一使用 `Flyway` 或 `Liquibase`，迁移脚本放在 `migrations/`，禁止手工改表
* `service` 层禁止拼接原生 SQL 字符串，确需复杂查询时使用 `@Query` 注解或 MyBatis XML 映射

### 7.3.5 测试

* 单元测试使用 `JUnit 5` + `Mockito`，类名以 `xxxTest` 结尾，与被测类包路径一致放在 `src/test/java`
* 集成测试使用 `@SpringBootTest` + `Testcontainers` 启动真实数据库，类名以 `xxxIT` 结尾
* 断言库统一使用 `AssertJ`，禁止裸用 `assertTrue` 堆砌复杂判断

### 7.3.6 服务级 Makefile 建议目标

```makefile
run              # mvn spring-boot:run
test             # mvn test
lint             # mvn spotless:check
build            # mvn clean package -DskipTests
docker-build     # docker build -t payment-service .
migrate-up       # mvn flyway:migrate
migrate-down     # mvn flyway:undo
```

## 7.4 Python 服务结构

```bash
services/ai-service/
├── app/
│   ├── api/               # 路由入口（FastAPI router / Flask blueprint）
│   ├── services/           # 业务逻辑编排
│   ├── repositories/        # 数据访问层
│   ├── models/                # ORM 模型 / Pydantic Schema
│   ├── core/                    # 配置加载、依赖注入、异常处理
│   └── main.py
├── tests/
│   ├── unit/
│   └── integration/
├── configs/
├── migrations/
├── requirements.txt        # 或 pyproject.toml（推荐 Poetry）
├── .flake8 / pyproject.toml
├── Dockerfile
├── Makefile
└── README.md
```

### 7.4.1 分层依赖方向

```text
api → services → repositories → models
```

* `api` 层只做请求解析（推荐 `FastAPI` + `Pydantic` 做参数校验）、调用 `services`、返回响应
* `services` 通过依赖注入（FastAPI `Depends` 或显式构造函数传参）使用 `repositories`，不直接引用 ORM Session
* 跨服务共享代码下沉到 `packages/`，`app/` 内部禁止循环引用

### 7.4.2 命名与代码风格

* 统一使用 **Python 3.11+**，模块名、文件名使用 snake_case（如 `order_service.py`）
* 类名使用 **PascalCase**，函数/变量使用 **snake_case**，常量使用 **UPPER_SNAKE_CASE**
* 强制使用类型注解（Type Hints），CI 中执行 `mypy` 静态类型检查
* 统一使用 `black` 格式化、`isort` 整理 import、`flake8`（或 `ruff`）做 lint，CI 中执行 `ruff check` / `black --check`
* 优先使用 `FastAPI`（新服务默认选型），历史 `Flask` 服务维持现状，不强制迁移

### 7.4.3 配置与依赖注入

* 配置统一使用 `pydantic-settings`（`BaseSettings`）从环境变量 / `.env.<env>` 加载，禁止直接 `os.environ.get` 散落各处
* 依赖注入统一使用 FastAPI 的 `Depends`，`services` 与 `repositories` 通过构造函数接收依赖，便于测试时替换
* 敏感配置通过环境变量或 Secret 管理注入，禁止提交到 `requirements.txt` 同级的配置文件

### 7.4.4 数据访问

* 统一使用 `SQLAlchemy`（同步用 `SQLAlchemy Core/ORM`，异步用 `SQLAlchemy 2.0 async` 或 `Tortoise ORM`，团队内统一选型）
* 数据库迁移统一使用 `Alembic`，迁移脚本放在 `migrations/`，禁止手工改表
* `services` 层禁止直接执行原生 SQL 字符串拼接，确需原生 SQL 时使用参数化查询并封装在 `repositories/`

### 7.4.5 测试

* 单元测试使用 `pytest`，文件命名 `test_*.py`，放在 `tests/unit/`，与被测模块结构对应
* 集成测试放在 `tests/integration/`，通过 `httpx.AsyncClient` 驱动 FastAPI 应用，数据库依赖通过 `testcontainers` 启动
* Mock 统一使用 `unittest.mock` 或 `pytest-mock`，禁止单元测试中发起真实网络请求
* 覆盖率要求纳入 CI（`pytest --cov`），核心业务模块建议不低于团队约定阈值

### 7.4.6 服务级 Makefile 建议目标

```makefile
run              # uvicorn app.main:app --reload
test             # pytest --cov=app
lint             # ruff check . && black --check .
build            # docker build -t ai-service .（Python 无独立编译产物，构建即打包镜像）
docker-build     # docker build -t ai-service .
migrate-up       # alembic upgrade head
migrate-down     # alembic downgrade -1
```

## 7.5 .NET Core 服务结构

统一采用 **Clean Architecture / 分层架构** 组织代码，解决方案（`.sln`）与项目（`.csproj`）命名统一使用 **PascalCase**，与 .NET 生态惯例保持一致；仅目录顶层（`services/xxx-service`）沿用仓库统一的 kebab-case。


### 7.5.1 项目分层依赖方向

依赖关系必须单向指向领域层，禁止反向依赖：

```text
Api → Application → Domain
Infrastructure → Application → Domain
```

* `Domain` 不依赖任何其他层，不引用 EF Core、ASP.NET Core 等框架包
* `Application` 仅依赖 `Domain`，通过接口（如 `IOrderRepository`）解耦基础设施实现
* `Infrastructure` 实现 `Application` 定义的接口，负责数据库、缓存、消息队列等具体实现
* `Api` 仅做请求编排（参数校验、调用 Application、返回响应），不写业务逻辑

### 7.5.2 命名与代码风格

* 命名空间与项目名保持一致：`<Domain>Service.<Layer>`，如 `OrderService.Application`
* 类名、方法名使用 **PascalCase**；参数、局部变量使用 **camelCase**；私有字段使用 `_camelCase`
* 接口统一以 `I` 前缀命名，如 `IOrderRepository`、`IPaymentClient`
* 异步方法统一以 `Async` 结尾，如 `GetOrderAsync`
* 统一使用 `.editorconfig` 约束代码风格，并在 CI 中执行 `dotnet format --verify-no-changes`
* `Program.cs` 禁止使用顶级语句（top-level statements），统一写成显式的 `public class Program { public static void Main(string[] args) { ... } }`，便于集成测试通过 `WebApplicationFactory<Program>` 引用入口类型，也让启动逻辑的调用栈在调试时更清晰

### 7.5.3 配置与依赖注入

* 使用 `appsettings.{Environment}.json` 管理分环境配置，环境名遵循本规范第 14 章（`dev / test / staging / prod`）
* 禁止使用 ASP.NET Core 内置的 `IWebHostEnvironment.IsDevelopment()` / `IsStaging()` / `IsProduction()`：这组方法只识别固定字符串 `"Development"` / `"Staging"` / `"Production"`，与本规范的 `dev/test/staging/prod` 命名不一致（且没有 `test` 对应项）。判断当前环境统一用 `string.Equals(app.Environment.EnvironmentName, "dev", StringComparison.OrdinalIgnoreCase)` 这种直接比较方式
* 禁止在 `appsettings.json` 中存放真实密钥，敏感配置通过环境变量或 Secret 管理注入
* 服务注册统一在 `Program.cs` 或按层拆分的 `ServiceCollectionExtensions`（如 `AddApplicationServices()`、`AddInfrastructureServices()`）中完成，禁止分散在各处手动 `new`

### 7.5.4 数据访问

* 统一使用 **EF Core**（或团队约定的 ORM）作为运行时数据访问层，仓储实现放在 `Infrastructure/Repositories/`
* 数据库迁移**不使用** EF Core 自带的迁移机制（`dotnet ef migrations add` / `dotnet ef database update`），统一手写原生 SQL 迁移脚本放在服务根目录 `migrations/`，与 Go 服务共用同一套迁移工具（`golang-migrate`），保证跨语言服务的迁移方式、命令、CI 接入一致
* EF Core 的 `DbContext` / 实体配置（`Infrastructure/Persistence/Configurations/`）与 `migrations/` 下的 SQL 脚本描述的是同一份表结构，两者不再由同一份迁移文件自动保证同步——改动表结构时必须同时更新 SQL 脚本和 EF 实体配置，Review 时需交叉核对
* 禁止在 `Domain` 层直接引用 `DbContext`

### 7.5.5 测试

* 单元测试项目命名为 `<Service>.UnitTests`，集成测试项目命名为 `<Service>.IntegrationTests`
* 单元测试统一使用 `xUnit` + `Moq`（或团队统一选型），断言库统一使用 `Shouldly`（开源 MIT 许可证，避免 `FluentAssertions` 8.x 起对商业公司收费的授权风险）
* 集成测试通过 `WebApplicationFactory` 或 `Testcontainers` 启动真实依赖，禁止依赖开发者本机环境

### 7.5.6 服务级 Makefile 建议目标

```makefile
run              # dotnet run --project src/OrderService.Api
test             # dotnet test
build            # dotnet build -c Release
publish          # dotnet publish -c Release
docker-build     # docker build -t order-service .
migrate-up       # migrate -path migrations -database $(DB_URL) up
migrate-down     # migrate -path migrations -database $(DB_URL) down 1
```

# 16. API 规范

## 16.1 路径规范

统一采用 REST 风格：

```http
GET    /api/v1/users
GET    /api/v1/users/{id}
POST   /api/v1/users
PUT    /api/v1/users/{id}
DELETE /api/v1/users/{id}
```

路径格式：

```text
/api/{version}/{resource}
```

示例：

* `/api/v1/users`
* `/api/v1/orders`
* `/api/v1/products`
* `/api/v1/payments`

禁止使用：

* `/getUser`
* `/createOrder`
* `/deleteProduct`

## 16.2 健康检查接口

所有服务必须提供：

```http
GET /health
```

可选扩展：

* `/ready`
* `/metrics`

`/health`（及 `/ready`、`/metrics`）不加路由前缀：K8s 探针与监控直接访问 Pod，不经过网关，因此不受 16.3 节路由前缀规则约束。

## 16.3 网关路由前缀规范

所有微服务通过统一的 K8s 服务网关对外暴露，网关按 **host + 路由前缀** 转发到对应服务，且**不 strip 前缀**（原样转发给后端）。因此每个服务必须自带路由前缀，服务内部路由注册时需包含该前缀，保证网关转发路径与服务内部路由完全一致。

### 16.3.1 前缀格式

```text
/<domain>-service
```

与服务名（第 6 章）保持一致，直接使用完整服务名作为前缀，不做缩写或省略。

示例：

* `sso-service` → `/sso-service`
* `order-service` → `/order-service`
* `payment-service` → `/payment-service`

### 16.3.2 完整路径格式

```text
/<domain>-service/api/{version}/{resource}
```

示例：

* `/sso-service/api/v1/auth/login`
* `/order-service/api/v1/orders`
* `/payment-service/api/v1/payments`

### 16.3.3 规则

* 路由前缀必须在服务内部路由层（如 Go 服务的 `internal/handler/router.go`）显式注册，不依赖网关做路径重写
* 前缀不 strip，服务内部路由、日志、链路追踪中看到的路径均带前缀，便于排查请求来源服务
* `/health`、`/ready`、`/metrics` 例外，不加前缀（见 16.2）
* 网关侧每新增一个服务，必须同步更新路由映射文档（见 `docs/route-mapping.md`），并在网关 Ingress/IngressRoute 资源中新增对应规则
* 禁止跨服务复用同一前缀，前缀与服务一一对应

### 16.3.4 CORS 跨域策略

CORS 统一在网关层配置（如 Traefik `Middleware`），服务自身不重复处理，避免网关与服务同时添加 `Access-Control-Allow-*` 响应头导致浏览器拒绝（重复 header）。

* 允许来源使用白名单，禁止使用 `*` 通配（尤其在需要携带 `Authorization` / Cookie 的接口上，通配符与 credentials 模式浏览器本身也不兼容）
* 白名单变更只需修改网关侧 CORS 配置并重新部署，服务代码无需改动、无需重新构建镜像

## 16.4 分页与排序参数规范

所有返回列表的分页接口（`GET` 且返回 `PagedResult`/等价分页结构的接口）必须支持指定排序字段，不得只支持固定排序。

### 16.4.1 查询参数

统一使用以下四个 query 参数，不允许自造命名（如 `order_by`、`sort_field`）：

| 参数 | 说明 | 默认值 |
| --- | --- | --- |
| `page` | 页码，从 1 开始 | `1` |
| `pageSize` | 每页条数 | `20`，上限 `200` |
| `sortBy` | 排序字段名（驼峰命名，对应响应体字段） | 由各资源自行约定（通常为 `id` 或创建时间） |
| `sortOrder` | 排序方向：`Asc` / `Desc` | `Asc` |

示例：

```http
GET /api/v1/orders?page=1&pageSize=20&sortBy=createdAt&sortOrder=desc
```

### 16.4.2 可排序字段白名单

`sortBy` 必须校验是否属于该资源显式声明的**可排序字段白名单**，禁止将请求参数直接拼接为 ORM/SQL 排序表达式（避免任意字段排序带来的信息泄露与 SQL 注入风险）。传入白名单之外的字段名，接口必须返回 400（业务校验异常），而不是静默忽略或报 500。

### 16.4.3 响应体格式

分页响应统一使用与第 8-9 章一致风格的结构：

```json
{
  "items": [ ... ],
  "page": 1,
  "pageSize": 20,
  "total": 137
}
```

字段名固定为 `items`/`page`/`pageSize`/`total`，各语言按自身命名惯例做大小写转换（如 Go/Java 用 `Items`、Node/Python 序列化为 `items`），但语义与结构必须一致。

### 16.4.4 实现要求

* 排序逻辑放在 Repository/数据访问层，Service/Application 层只负责校验 `sortBy` 是否在白名单内并解析出（字段名, 排序方向）二元组传给 Repository
* 每个资源的可排序字段白名单、默认排序字段由该资源自己声明，不跨资源共享同一份白名单
* 新增列表接口时必须同时支持 `sortBy`/`sortOrder`，不允许以"后续再加"为由先上线固定排序的分页接口

## 16.5 仅内部调用接口命名规范

仅供集群内其他服务直连调用、不经网关暴露、不面向外部客户端的接口（如服务间鉴权校验、内部数据查询/状态回写），路由与实现方法名都必须显式带上 `internal` 标识，与对外 REST API 在命名上一眼区分，防止误当成公开接口被网关路由或被外部文档引用。

### 16.5.1 路由规范

路径必须以 `/internal/` 开头，不带 16.3 节的 `/<domain>-service` 网关前缀（该前缀是给网关转发用的，内部接口不经网关，不需要）：

```text
/internal/{resource}
/internal/{resource}/{action}
```

示例：

* `GET /internal/auth/verify`
* `GET /internal/users/{id}`
* `PUT /internal/users/{id}/review`

### 16.5.2 方法命名规范

Handler/Controller 方法名、对应的 Service/Application 层方法名，必须以 `Internal` 为前缀或后缀（同一服务内保持一致，推荐后缀），不能与对外接口的方法命名混在一起靠路径区分：

| 语言 | 示例 |
| --- | --- |
| Go | `GetUserInternal`、`ApproveReviewInternal` |
| .NET | `GetUserInternalAsync`、`ApproveReviewInternalAsync` |
| Java | `getUserInternal`、`approveReviewInternal` |
| Node.js/TypeScript | `getUserInternal`、`approveReviewInternal` |
| Python | `get_user_internal`、`approve_review_internal` |

### 16.5.3 判定标准

一个接口是否属于"仅内部调用"，看它是否**专为服务间调用设计**，而非"当前只是恰好被内部服务调用"：

* 属于：服务间鉴权校验（如 `/internal/auth/verify`）、供其他服务查询/回写本服务私有数据且不打算对外开放的接口
* 不属于：服务自身对外的公开业务 API（如作业创建、订单创建），即使当前调用方只有集群内的其他服务——这类接口未来可能直接对外开放或被网关暴露，命名与路由维持对外 API 规范（16.1-16.3），只在鉴权层面按需加共享密钥等内部调用校验，不改路由和方法名

### 16.5.4 鉴权

仅内部调用接口默认信任"不经网关暴露"的网络边界是不够的，必须额外校验调用方身份（如集群内共享密钥、mTLS），具体机制不强制统一，但必须实现，不允许零校验直接放行。

### 16.5.5 集群内调用方式

服务间调用（无论是 16.5.1 定义的仅内部接口，还是服务自身对外 API 被集群内其他服务调用的场景）统一走 **K8s Service DNS 直连**，不经过网关：

```text
http://<service-name>.<namespace>.svc.cluster.local
```

`<service-name>`/`<namespace>` 对应目标服务 `deploy/k8s/services/<domain>-service/service.yaml` 里的 `metadata.name`/`metadata.namespace`。所有服务的 K8s Service HTTP 端口默认暴露为 `80`，因此服务间 HTTP 调用的 Base URL 默认不写端口；容器内部监听端口可继续使用 `8080` 等端口，由 Service `targetPort` 映射。示例：

```text
http://sso-service.default.svc.cluster.local
http://backend-job-service.default.svc.cluster.local
```

**原理**：K8s 集群自带 DNS（通常是 CoreDNS），每个 Pod 的 DNS 解析默认指向它，`<service-name>.<namespace>.svc.cluster.local` 会被解析为目标 `Service` 的 ClusterIP（虚拟 IP，非具体 Pod IP）；请求打到 ClusterIP 后由每个节点上的 kube-proxy 按 `Service.spec.selector` 转发到某个健康 Pod，全程在集群网络平面内完成，不经过任何反向代理或网关。这条路径与网关（如 Traefik）处理的"外部入口流量"是两条独立路径：网关只挂在集群对外暴露的 Ingress 上，管不到 Pod 间的直连流量。

**约束**：

* 调用方必须使用目标服务的 Service DNS 全名，不允许硬编码 Pod IP 或使用 `port-forward` 之类仅限本地调试的访问方式
* 目标服务必须已存在对应的 K8s `Service` 对象（`type: ClusterIP` 即可，不需要 `NodePort`/`LoadBalancer`），二者同属一个集群、同一 `namespace`（跨 namespace 需要网络策略放行）
* 网关的 ForwardAuth（见 16.3、[deploy/k8s/gateway/auth-middleware.yaml](deploy/k8s/gateway/auth-middleware.yaml)）**不覆盖**这条路径，因此被集群内直连调用的接口（无论是否为 16.5.1 定义的仅内部接口）必须自行实现调用方鉴权（见 16.5.4），不能假设"没经过网关=已被拦截未授权访问"
* 若集群启用了 `NetworkPolicy`，调用方与目标服务的 Pod 之间必须显式放行对应端口，否则 DNS 能解析但连接不通

# 24. 团队强制约定

以下规则应写入 `CONTRIBUTING.md` 并强制执行：

1. 所有目录统一使用 `kebab-case`
2. 所有业务服务必须放在 `services/`
3. 所有公共包必须放在 `packages/`
4. 服务命名必须使用 `<domain>-service`
5. 每个服务必须包含 `README.md`、`Dockerfile`、`Makefile`、`service.yaml`
6. 所有脚本必须放在 `scripts/`
7. 脚本命名必须使用“动词-对象”格式
8. 环境命名只允许 `dev / test / staging / prod`
9. 所有服务必须提供 `/health`
10. API 必须使用 `/api/v1/...`
11. 提交信息必须符合 Conventional Commits
12. 禁止提交生产密钥、密码、Token、证书等敏感信息
13. 新服务上线前必须补齐 README、基础测试、部署配置和服务元信息
